## Project Overview

Terraform + Ansible project that provisions a Kubernetes cluster on Azure (kubeadm-based, with Flannel CNI). VMs are bootstrapped via cloud-init; security hardening runs via Ansible post-provisioning. CI/CD uses GitHub Actions with manual approval gates before apply/destroy.

## Commands

### Terraform

```bash
terraform init                          # Initialize (downloads providers)
terraform init -backend=false           # Init without remote state (for local validation)
terraform fmt -check -recursive         # Check formatting
terraform fmt -recursive                # Fix formatting
terraform validate                      # Validate configuration
tflint --init && tflint -f compact      # Lint
terraform plan -out=tfplan              # Generate and save plan
terraform apply tfplan                  # Apply saved plan
terraform destroy                       # Destroy infrastructure
```

### Ansible

```bash
ansible-playbook -i inventory.ini ansible/playbooks/bootstrap-k8s.yml
ansible-playbook -i inventory.ini ansible/playbooks/harden-vms.yml
```

### Post-deployment verification

```bash
terraform output control_plane_public_ip
terraform output -raw kubeadm_join_command
# On the control plane VM:
sudo kubectl --kubeconfig=/etc/kubernetes/admin.conf get nodes -o wide
sudo kubectl --kubeconfig=/etc/kubernetes/admin.conf get pods -A
```

## Architecture

### Terraform modules

| File | Responsibility |
|------|---------------|
| `main.tf` | Provider config, resource group, random kubeadm bootstrap token |
| `network.tf` | VNet, subnet, NSG (SSH/6443/NodePort/ICMP rules) |
| `compute.tf` | Control plane + worker VMs, RSA-4096 SSH keypair, public IPs, NICs |
| `key_vault.tf` | Azure Key Vault storing the SSH private key |
| `state_storage.tf` | Azure Storage Account + container for remote tfstate |
| `variables.tf` | All input variables with defaults |
| `outputs.tf` | Public IPs, kubeconfig command, SSH key, join command |

### Cloud-init bootstrap flow

1. `cloud-init/control-plane.yaml` — installs containerd + kubeadm/kubelet/kubectl, runs `kubeadm init`, deploys Flannel CNI
2. `cloud-init/worker.yaml` — same container runtime install, then `kubeadm join` using token from `main.tf`

The kubeadm bootstrap token is generated by Terraform (`random_string`) and injected into both cloud-init files via `templatefile()`.

### CI/CD pipeline (`.github/workflows/`)

- **`terraform.yml`**: On PR → fmt/validate/tflint; on push → plan (saved as artifact); manual `workflow_dispatch` → apply or destroy (both behind GitHub Environment approval gates: `terraform-apply` / `terraform-destroy`)
- **`ansible-hardening.yml`**: Triggered after successful apply; runs `harden-vms.yml` against provisioned IPs

### Key variables

| Variable | Default | Notes |
|----------|---------|-------|
| `prefix` | `ACA` | Prepended to all resource names |
| `location` | `west europe` | Azure region |
| `worker_count` | `2` | Number of worker nodes |
| `vm_size` / `vm_size_wn` | `Standard_d2s_v3` | Min 2 vCPU / 4 GB RAM for k8s |
| `pod_network_cidr` | `10.244.0.0/16` | Must match Flannel default |

### Required GitHub Secrets

Azure SP: `ARM_CLIENT_ID`, `ARM_CLIENT_SECRET`, `ARM_SUBSCRIPTION_ID`, `ARM_TENANT_ID`  
Backend: `TFSTATE_RESOURCE_GROUP_NAME`, `TFSTATE_STORAGE_ACCOUNT_NAME`, `TFSTATE_CONTAINER_NAME`, `TFSTATE_KEY`

## Conventions

- Resource names follow the pattern `${var.prefix}-<resource-type>` (e.g. `ACA-vnet`, `ACA-cp-vm`)
- State is stored remotely in Azure Blob; never commit `terraform.tfstate`
- The SSH private key is only in Key Vault and Terraform outputs — never committed
- Kubernetes version is pinned in cloud-init; update both `control-plane.yaml` and `worker.yaml` together when upgrading
